Актуализированный аудит безопасности личного кабинета (май 2026, повторная проверка)

Фокус: backend + frontend ЛК, документы, кейсы, сообщения, контроль доступа и приватность.

========================================
КРИТИЧЕСКИЕ ПРОБЛЕМЫ (исправить в первую очередь)
========================================

1) ПУБЛИЧНАЯ РАЗДАЧА ФАЙЛОВ ИЗ КОРНЯ ПРОЕКТА (не только frontend)
- Где: backend/app.py -> route "/<path:requested_path>" (serve_site_files).
- Риск: можно получить чувствительные файлы напрямую по HTTP (например БД, backend-исходники, конфиги), если путь существует в project root.
- Что сделать:
  - разрешить раздачу только белого списка публичных директорий (например frontend/, img/, css/, js/, статические html);
  - явно запретить backend/, database/, .env*, storage/, scripts/ и служебные файлы;
  - по возможности вынести БД/секреты вне web-root.

========================================
ВЫСОКИЙ ПРИОРИТЕТ
========================================

2) LOGIN БЕЗ ПРОВЕРКИ EMAIL VERIFICATION
- Где: backend/routes/auth.py -> /login.
- Риск: аккаунт может авторизоваться без подтверждения email (если это не бизнес-требование).
- Что сделать:
  - до логина проверять флаг/состояние верификации;
  - либо зафиксировать это как осознанную политику и обновить продуктовые требования.

3) ИЗБЫТОЧНЫЕ ПРАВА В ACL КЕЙСА (расхождение с ACL workspace/documents)
- Где: backend/routes/lk.py -> _viewer_may_access_client_case_data().
- Риск: часть staff-ролей с правами коммуникации получает доступ к кейсам шире, чем доступ к документам/рабочему пространству этого же клиента.
- Что сделать:
  - унифицировать ACL кейса с централизованной проверкой workspace-доступа;
  - убрать "широкие" bypass-ветки доступа для ролей без явной привязки к клиенту.

4) СМЕНА РОЛИ ПО NUMERIC USER_ID БЕЗ ДОСТАТОЧНОЙ ПРОВЕРКИ ВИДИМОСТИ ЦЕЛИ
- Где: backend/routes/user.py -> change_user_role().
- Риск: пользователь с правом назначения ролей может менять роль цели, которую не должен администрировать (по знанию внутреннего id).
- Что сделать:
  - добавить проверку "может ли actor управлять именно target";
  - запретить изменение ролей пользователей равного/более высокого уровня.

5) STORED XSS В СПИСКЕ КЛИЕНТОВ (ЛК)
- Где: frontend/js/clients.js (renderUsersTable/renderUsersCards/buildUserAvatarHtml).
- Риск: данные профиля (name/email/avatar URL) вставляются в innerHTML без экранирования; XSS в интерфейсе staff.
- Что сделать:
  - экранировать все текстовые поля;
  - санитизировать URL аватара (allowlist схем/доменов или backend proxy);
  - избегать небезопасной генерации HTML-шаблонов для пользовательских полей.

========================================
СРЕДНИЙ ПРИОРИТЕТ
========================================

6) ДОПОЛНИТЕЛЬНЫЕ XSS-ТОЧКИ В ЛК
- Где:
  - frontend/js/case.js (name/description в запросах документов, avatar в блоке менеджера);
  - frontend/js/documents.js (avatar редактора в истории);
  - frontend/js/progress-tracker.js (title/description этапов через innerHTML).
- Риск: выполнение произвольного JS через сохраненные пользовательские/админские поля.
- Что сделать:
  - унифицированный escape/sanitize слой для всех HTML-вставок;
  - по возможности заменить innerHTML на textContent + безопасное построение DOM.

7) USER ENUMERATION НА РЕГИСТРАЦИИ И СМЕНЕ EMAIL
- Где:
  - backend/routes/auth.py -> register() (409 "already exists");
  - backend/routes/lk.py -> update_current_user_profile() (409 "email already exists").
- Риск: подтверждение существования email в системе.
- Что сделать:
  - унифицировать ответы (без раскрытия факта существования учетной записи);
  - оставить детали только в server logs.

8) ИЗМЕНЕНИЕ EMAIL БЕЗ ПОВТОРНОЙ ВЕРИФИКАЦИИ
- Где: backend/routes/lk.py -> update_current_user_profile().
- Риск: при компрометации сессии атакующий может перепривязать email.
- Что сделать:
  - требовать re-verification для нового email;
  - до подтверждения хранить pending_email и не заменять основной email.

9) УТЕЧКА ВНУТРЕННИХ ДЕТАЛЕЙ ЧЕРЕЗ str(e) В API-ОШИБКАХ
- Где: backend/routes/messages.py, backend/routes/documents.py, backend/routes/application_progress.py и др.
- Риск: раскрытие внутренних путей/трассировки/SQL в ответах клиенту.
- Что сделать:
  - отдавать клиенту унифицированное "internal error";
  - писать детали только в backend logs.

10) RATE LIMIT ОБХОДИМ ПОДМЕНОЙ X-FORWARDED-FOR
- Где: backend/app.py, backend/routes/auth.py (получение IP из заголовка без модели доверенного proxy).
- Риск: обход лимитов brute-force при прямой подстановке заголовка.
- Что сделать:
  - доверять X-Forwarded-For только от известного reverse proxy;
  - иначе использовать request.remote_addr.

11) IN-MEMORY RATE LIMITER НЕМАСШТАБИРУЕМ И НЕУСТОЙЧИВ К RESTART
- Где: backend/utils/rate_limiter.py.
- Риск: лимиты сбрасываются при рестарте и не делятся между процессами/репликами.
- Что сделать:
  - перенести limiter в Redis (или другой shared storage).

12) /api/users/search УЖЕ ОГРАНИЧЕН ДО display_id, НО НЕТ ОТДЕЛЬНОГО RATE LIMIT
- Где: backend/routes/messages.py -> /api/users/search.
- Риск: перебор display_id авторизованным пользователем.
- Что сделать:
  - добавить отдельный rate limit (IP + user);
  - аудит событий поиска.

13) ХРАНЕНИЕ ПРОФИЛЯ В localStorage (PII) + legacy numeric userId в URL
- Где:
  - frontend/js/lk.js, frontend/js/profile.js (кэш профиля в localStorage);
  - frontend/js/case.js, frontend/js/documents.js, frontend/js/clients.js, frontend/js/dashboard.js (fallback/query с userId).
- Риск: утечка PII при XSS, лишняя экспозиция внутреннего id в URL/history/logs.
- Что сделать:
  - минимизировать данные в localStorage;
  - перейти полностью на публичный display_id в URL/навигации.

14) UPLOAD PIPELINE УСИЛЕН ЧАСТИЧНО (нет AV/sanitization)
- Где: backend/services/file_service.py.
- Риск: загрузка потенциально вредоносных файлов (особенно архивов) без AV-сканирования.
- Что сделать:
  - добавить AV scan (например ClamAV);
  - ужесточить MIME policy, снизить доверие к application/octet-stream;
  - для отдельных типов рассмотреть sanitize/re-encode.

15) CSP ЕСТЬ, НО ОСЛАБЛЕН ('unsafe-inline', широкий img-src)
- Где: backend/app.py -> Content-Security-Policy.
- Риск: защита от XSS слабее, чем может быть.
- Что сделать:
  - перейти на nonce/hash для inline;
  - сузить внешние источники (script/style/img/font) до фактически нужных.

========================================
НИЗКИЙ ПРИОРИТЕТ / HARDENING
========================================

16) VERBOSE console.log С PII В FRONTEND
- Где: frontend/js/case.js и др.
- Риск: утечки в консоль/скриншоты/сессии поддержки.
- Что сделать: убрать debug-логи из production-сборки.

17) target="_blank" без rel="noopener"
- Где: frontend/js/case.js (формирование ссылок на скачивание).
- Риск: tabnabbing.
- Что сделать: добавить rel="noopener noreferrer".

18) КОНСИСТЕНТНОСТЬ SUPPORT_USER_ID (hardcoded во frontend)
- Где: frontend/js/chat.js, dashboard.js, profile.js, lk-i18n.js.
- Риск: рассинхронизация окружений.
- Что сделать: получать значение только из backend API.

========================================
УЖЕ ИСПРАВЛЕНО (по сравнению с предыдущей версией файла)
========================================

- /storage больше не публичный: есть auth + ACL проверка.
- Утечка file_path/archive_url в API и frontend существенно снижена:
  - выдача через защищенные download endpoints.
- Токен перенесен из localStorage в HttpOnly cookie.
- Добавлены security headers (CSP/X-Frame-Options/nosniff/Referrer-Policy/Permissions-Policy/HSTS на HTTPS).
- CORS переведен на whitelist (без отражения любого Origin).
- SECRET_KEY усилен для production (fail-fast + минимальная длина).
- Добавлен rate limiting для auth/upload/messages.
- /api/users/search ограничен форматом публичного display_id и минимальной выдачей.
- Debug запуск в точке входа переведен под env-политику; production через WSGI.

========================================
РЕКОМЕНДУЕМЫЙ ПОРЯДОК ИСПРАВЛЕНИЙ
========================================

Шаг 1 (немедленно):
- закрыть публичный catch-all доступ к чувствительным файлам project root (пункт 1);
- закрыть XSS в clients.js и других подтвержденных innerHTML-точках (пункты 5, 6);
- выровнять ACL кейса/ролей (пункты 3, 4).

Шаг 2:
- внедрить email-verification gate при login и безопасную смену email (пункты 2, 8);
- анти-enumeration ответы + лимит на users/search (пункты 7, 12);
- убрать выдачу str(e) в API (пункт 9).

Шаг 3:
- перейти на shared rate limiter (Redis) и корректную proxy-модель IP (пункты 10, 11);
- снизить поверхность утечки данных на frontend (пункт 13);
- усилить upload/CSP hardening (пункты 14, 15);
- закрыть low-priority hygiene пункты (16-18).

========================================
КРАТКИЙ ВЫВОД
========================================

Ключевой текущий риск ЛК: возможность получить чувствительные файлы через универсальную раздачу из project root и наличие подтвержденных XSS-точек во frontend. Ранее отмеченные базовые hardening-пункты (cookies, security headers, CORS, SECRET_KEY, rate limiting, debug mode, users/search) в значительной части уже закрыты, но проекту требуется второй этап hardening и унификация ACL.
